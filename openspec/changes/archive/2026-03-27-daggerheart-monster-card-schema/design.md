## Context

Каталог уже собирается на GitHub Pages; стек — Vite, React, Mantine. Пример карточки задаёт визуальную иерархию: имя, тир, краткое описание, мотивы и тактики, таблица статов, строка опыта, блок FEATURES с именованными умениями и типами.

## Goals / Non-Goals

**Goals:**

- Однозначная схема полей в YAML, отражающая секции карточки.
- Валидация на сборке с понятными сообщениями.
- UI: `Card`/`Paper` (Mantine), секции с типографикой (имя крупно, tier, flavor курсивом, stats в одной строке с разделителями `|`, фичи — заголовок + тип + текст).

**Non-Goals:**

- Полный парсинг Markdown внутри описаний фич в первой итерации (достаточно текстового блока с возможной последующей доработкой).
- Импорт PDF/автогенерация из других систем.

## Decisions

| Решение | Выбор | Rationale |
|--------|--------|-----------|
| Имена полей | Английские ключи в YAML (`tier`, `flavor`, `stats`, …) | Совпадение с терминами и стабильность API |
| `tier` | Одна строка, например `Tier 1 Solo` | Как на примере карточки |
| `stats.thresholds` | Строка `"8/15"` | Компактно; при необходимости позже распарсить |
| `stats.attack` | Объект: `atk`, `name`, `range`, `damage` | Соответствует строке ATK \| имя: дальность \| урон |
| `features[]` | `name`, `type` (`passive` \| `action` \| `reaction`), опционально `cost`, обязательно `description` | Соответствует «Name (3) - Passive: …» |

## Risks / Trade-offs

- **[Risk]** Ломающее изменение для старых YAML. **→** Миграция примеров и явная документация в README.
- **[Trade-off]** Жёсткая схема вместо свободных полей — меньше гибкости, но выше согласованность с карточкой.

## Migration Plan

1. Зафиксировать схему в `openspec/specs` (после архива change).
2. Обновить YAML и скрипт валидации.
3. Обновить типы и UI.
4. Проверить `npm run build` и превью с base path.
