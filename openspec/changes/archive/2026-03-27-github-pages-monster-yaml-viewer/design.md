## Context

Репозиторий `dh-monster-lib` пока без приложения: нужен публичный просмотр каталога монстров. Данные живут в Git как YAML; хостинг — только статический (GitHub Pages), без серверного API. Ограничения: CORS при прямом fetch YAML с `raw.githubusercontent.com`, корневой URL Pages вида `https://<user>.github.io/<repo>/` (base path).

## Goals / Non-Goals

**Goals:**

- Статическая сборка фронтенда, пригодная для GitHub Pages.
- Один согласованный формат YAML и предсказуемое расположение файлов в репо.
- Список монстров и отображение полей; понятное сообщение при сбое загрузки.
- Автоматическая публикация через GitHub Actions при изменениях в основной ветке.

**Non-Goals:**

- Бэкенд, авторизация, редактирование монстров в UI.
- Полноценный i18n и тёмная тема в первой итерации (можно заложить семантическую разметку без обязательной реализации).
- Сложный поиск/фильтрация сверх простого текстового фильтра по имени (опционально позже).

## Decisions

| Решение | Выбор | Рationale | Альтернативы |
|--------|--------|-----------|--------------|
| Парсинг YAML в браузере | Сборочный шаг: YAML → JSON в `public/` (или `dist/`) | Нативный `fetch` JSON без CORS-проблем к тому же origin после деплоя; не тянем тяжёлый парсер в runtime если не нужно | Прямой fetch raw YAML: упирается в CORS и размер бандла (js-yaml) |
| Стек UI | Vite + TypeScript + лёгкий UI (vanilla или Preact) | Быстрая сборка, предсказуемый output, малый бандл для статики | Next.js: избыточен для чисто статического каталога на Pages |
| Base path | `base` в Vite = `/<repo>/` через env `GITHUB_REPOSITORY` или переменная в CI | GitHub Pages project sites требуют подпуть | Только user/org Pages на `/`: не подходит для типичного `github.io/repo` |
| Данные | Каталог `data/monsters/*.yaml` или один `data/monsters.yaml` + схема с массивом `monsters` | Версионируемо, диффы в PR понятны | БД: вне scope |

## Risks / Trade-offs

- **[Risk]** Неверный `base` в CI → сломанные ассеты и fetch. **→** Проверка в workflow через `VITE_BASE_URL` / документирование локального `npm run build -- --base=/repo/`.
- **[Risk]** Расхождение схемы YAML и UI. **→** Типы TypeScript, общие с контрактом JSON; при необходимости лёгкая валидация на этапе сборки.
- **[Trade-off]** Генерация JSON при сборке добавляет шаг: зато стабильный runtime и проще тесты.

## Migration Plan

1. Добавить структуру данных и пример YAML.
2. Реализовать сборку сайта и скрипт конвертации YAML→JSON (или единый импорт в Vite).
3. Включить workflow Pages: `actions/upload-pages-artifact` + `actions/deploy-pages` (или `peaceiris/actions-gh-pages` с ограничением на ветку `gh-pages`).
4. В настройках репозитория: Source = GitHub Actions. Откат — откат коммита или отключение workflow.
