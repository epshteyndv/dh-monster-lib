## Context

Сейчас модалка «Выберите монстра» живёт в `App.tsx`: `Modal`, `ScrollArea`, список `Button` по `monsters`. Поля `Monster.tier` (number) и `Monster.role` (`MonsterRole`) уже есть в типах.

## Goals / Non-Goals

**Goals:**

- **Фильтры:** независимые мультивыборы по **tier** и **role**. Варианты для tier — уникальные значения из текущего каталога (или отсортированный набор целых), для role — уникальные `MonsterRole` из каталога или полный enum — предпочтительно **только те роли и tier, что встречаются в данных**, чтобы не показывать пустые опции.
- **Семантика:** показывать монстра, если `(selectedTiers пусто ИЛИ tier ∈ selectedTiers)` **И** `(selectedRoles пусто ИЛИ role ∈ selectedRoles)`.
- **Компонент:** вынести модалку в `src/components/AddMonsterModal.tsx` (или `EncounterAddMonsterModal.tsx`) с пропсами: `opened`, `onClose`, `monsters`, `onPick(monster)`.

**Non-Goals:**

- Поиск по имени, сортировка, сохранение фильтров в URL.
- Серверный фильтр.

## Decisions

1. **Mantine:** `MultiSelect` или `Checkbox.Group` / группы `Chip` — выбрать то, что лучше ложится на «несколько вариантов» и a11y; значения tier как строки в UI при необходимости (`String(t)`).

2. **Пустой результат:** показать короткое сообщение («Нет монстров по фильтру»), не падать.

3. **Состояние фильтра:** локальный `useState` внутри компонента модалки; при закрытии модалки — сброс фильтров **или** сохранение до следующего открытия — выбрать **сброс при закрытии** для предсказуемости (зафиксировать в реализации).

## Risks / Trade-offs

- **[Risk]** Слишком длинный список tier/role — **Mitigation:** скролл или компактный `MultiSelect`.
