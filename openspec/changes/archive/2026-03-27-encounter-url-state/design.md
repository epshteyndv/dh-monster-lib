## Context

Энкаунтер: `EncounterEntry[]` с `instanceId` + `monster`. Для URL достаточно упорядоченного списка **`monster.id`** из каталога (дубликаты — повторы в массиве). После загрузки `monsters.json` каждый `id` превращается обратно в полный объект; `instanceId` генерируются заново (`crypto.randomUUID()`), что допустимо для сессии.

## Goals / Non-Goals

**Goals:**

- **Формат:** JSON с полем версии, например `{ "v": 1, "ids": ["a", "b", "a"] }`, затем UTF-8 → **base64url** (без `+`, `/`, padding по необходимости согласно стандартной практике в браузере).
- **Параметр:** один ключ query, например `enc` (коротко), значение — base64url строка.
- **Инициализация:** после успешной загрузки каталога: если в URL есть `enc`, декодировать и **заменить** текущий энкаунтер разрешёнными записями; неизвестные `id` пропускать или игнорировать целиком — зафиксировать в реализации (предпочтительно: пропускать только битые id, остальное применять).
- **Запись:** при любом изменении энкаунтера обновлять query через **`history.replaceState`**, чтобы не засорять «Назад» и не терять base path (`import.meta.env.BASE_URL`).

**Non-Goals:**

- Синхронизация с `localStorage`.
- Сжатие gzip в URL.

## Decisions

1. **replaceState vs push** — только `replace` при программном обновлении энкаунтера.

2. **Порядок применения** — сначала `fetchCatalog`, затем применение URL, чтобы резолвить id.

3. **Лимит длины** — при превышении разумного порога (например 32k) не писать в URL или усечь с предупреждением в консоли — в tasks как проверка границы.

## Risks / Trade-offs

- **[Risk]** Очень длинный URL — **Mitigation:** документировать лимит; для типичного энкаунтера достаточно.

## Migration Plan

Чисто клиентский деплой; старые ссылки без `enc` ведут себя как сейчас (пустой энкаунтер).
