## Why

Нужен единый каталог монстров в репозитории (YAML), который можно версионировать вместе с кодом и публиковать как статический сайт на GitHub Pages без бэкенда. Сейчас такого канала нет: данные и публичный просмотр не зафиксированы в спецификации.

## What Changes

- Добавить статическое веб-приложение, собираемое для деплоя на GitHub Pages (корень сайта или подпуть `/<repo>/`).
- Хранить список монстров в репозитории в виде одного или нескольких YAML-файлов с согласованной схемой полей.
- На сайте: загрузка данных (fetch к YAML или к сгенерированному JSON на этапе сборки), список/карточки монстров, базовая навигация и читаемое отображение полей.
- Добавить workflow/настройки GitHub Actions (или эквивалент) для сборки и публикации артефакта в GitHub Pages при push в основную ветку.

## Capabilities

### New Capabilities

- `monster-yaml-data`: Формат YAML-монстров, расположение файлов в репозитории, минимальный набор полей и правила расширения.
- `github-pages-monster-ui`: Статический UI на GitHub Pages: загрузка данных, список и детальный/карточный вид, ограничения клиентского рантайма (CORS, base path), доступность и базовая обработка ошибок загрузки.

## Impact

- Новый фронтенд-проект (зависимости, lockfile, скрипты сборки).
- CI/CD: GitHub Actions и включение GitHub Pages в настройках репозитория.
- Репозиторий: каталог с YAML (и при необходимости пример/фикстуры), возможно генерация JSON при сборке для упрощения парсинга в браузере.
